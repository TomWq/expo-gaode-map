**V3 目标**
`v3` 的核心不是把 4 个包合并，而是把 4 个包统一到一套架构下，同时保留你现有的现实约束：

- `expo-gaode-map` 和 `expo-gaode-map-navigation` 继续分包，且仍然互斥安装
- `expo-gaode-map-search` 继续独立，避免不用时增加原生包体积
- `expo-gaode-map-web-api` 继续独立，作为纯 JS 数据层
- 重点解决的是 TS 层双维护、初始化分散、类型模型分裂、搜索来源割裂

一句话定义：
`v3 = 4 个独立发行包 + 1 套共享领域层 + 2 套原生 runtime + 1 套统一用户层协议`

**V3 包定位**
1. `expo-gaode-map`
地图核心包。负责地图渲染、定位、覆盖物、相机控制、几何计算。

2. `expo-gaode-map-navigation`
导航一体化包。负责导航 SDK、导航地图、导航状态机、路线引导、导航 UI。

3. `expo-gaode-map-search`
原生搜索 provider。只在用户显式安装时提供原生搜索、输入提示、逆地理编码等能力。

4. `expo-gaode-map-web-api`
纯 JS 数据 provider。负责 Web API 请求、缓存、重试、归一化、跨平台能力。

**V3 核心架构**
建议内部拆成 5 层。

1. `domain`
纯类型和纯数据模型，不依赖 React，不依赖原生。
内容包括：
- `LatLng`、`Bounds`、`POI`、`AOI`
- `Route`、`RouteStep`、`TransitPlan`
- `LocationPermissionState`
- `PrivacyState`
- `GaodeErrorCode`
- 标准事件 payload

2. `contracts`
统一接口协议层。
内容包括：
- `MapRuntimeAdapter`
- `SearchProvider`
- `GeocodeProvider`
- `RouteProvider`
- `LocationProvider`
- `SnapshotProvider`

3. `shared-react`
React/TS 公共用户层。
内容包括：
- 通用 `MapView` facade
- 通用 overlay props 和 hooks
- `RouteOverlay`、`AreaMaskOverlay`、`useRoutePlayback`
- 事件归一化
- controller/ref API

4. `runtime-core`
只服务 `expo-gaode-map`，接地图 SDK 的原生桥接。

5. `runtime-navigation`
只服务 `expo-gaode-map-navigation`，接导航 SDK 自带地图和导航能力。

这样做的关键价值是：
- 原生层仍然分开
- 用户层 API 可以尽量统一
- overlay 和 hooks 不再在两个包里复制维护

**建议目录结构**
可以按这个方向重组，但不一定一次到位：

```text
packages/
  core/
    src/
      runtime/
      bridge/
      entry/
  navigation/
    src/
      runtime/
      bridge/
      entry/
  search/
    src/
      provider/
      bridge/
      entry/
  web-api/
    src/
      provider/
      transport/
      normalizers/
      entry/

internal/
  domain/
  contracts/
  shared-react/
  shared-utils/
  shared-tests/
```

如果你不想新增 `internal/` workspace，也可以先放在 `packages/shared-*` 里。

**V3 要统一的东西**
这是大版本最值钱的部分。

1. 统一类型模型
现在四个包应该逐步只认一套领域类型。
例如：
- `RoutePoint`
- `NormalizedPOI`
- `MapPoi`
- `CameraUpdate`
- `HeadingUpdate`
- `PermissionStatus`

2. 统一错误模型
现在每包都有自己的错误语义，大版本应该统一成：
- 错误码
- 错误分类
- 是否可重试
- 是否用户配置问题
- 是否隐私/权限问题
- 是否 provider 问题

3. 统一初始化模型
建议 `v3` 不再让每个包各自维护松散全局状态，而是引入统一 runtime/config。

建议目标形态：
```ts
const runtime = createGaodeRuntime({
  webKey,
  privacyConfig,
  platform: 'core' | 'navigation',
});
```

但对外仍然可以保留兼容的：
```ts
ExpoGaodeMapModule.initSDK(...)
```

只是内部都走统一 runtime。

4. 统一 provider 模型
`search` 和 `web-api` 不能继续只是“两个并列能力集合”，应该变成同一类抽象的不同实现。

建议协议：
```ts
interface SearchProvider
interface GeocodeProvider
interface RouteProvider
```

然后：
- `expo-gaode-map-search` 导出 `createNativeSearchProvider()`
- `expo-gaode-map-web-api` 导出 `createWebSearchProvider()`、`createWebRouteProvider()`

5. 统一事件流
地图、导航、定位、搜索的事件最好统一一套约定：
- 明确实时事件
- 明确状态事件
- 明确完成事件
- 明确错误事件
- 明确取消语义

**四个包在 V3 的升级重点**

**1. core**
重点不是加更多组件，而是变成稳定的地图 runtime。

建议：
- 保留现有 API 形状
- 抽出 shared overlay 和 shared hook
- 原生桥接统一命令模型
- 把几何计算和 camera 逻辑继续纯 TS 化
- 把 MapView imperative API 向 controller 过渡

V3 卖点：
- 更稳定的地图 facade
- 更少双份实现
- 更强的事件一致性

**2. navigation**
重点不是“复制一套地图”，而是“导航平台化”。

建议：
- 地图部分接 shared facade
- 导航部分单独保留 runtime
- 拆出 headless navigation controller
- `NaviView` 变成默认 UI，而不是唯一 UI 入口
- 公交 fallback、预览路线、独立算路统一进 provider/route model

V3 卖点：
- 导航地图能力继续完整
- 但导航状态与 UI 解耦
- 以后支持自定义导航 UI 会容易很多

**3. search**
重点是从“函数集合”升级到“可选原生 provider”。

建议：
- 不再强调“这是一个搜索包”
- 改成“这是原生搜索实现”
- 导出 provider 工厂
- 标准化返回模型
- 明确与 web provider 的对齐字段

V3 卖点：
- 安装即获得原生搜索能力
- 不安装就没有体积负担
- 与 web provider 可替换

**4. web-api**
重点是升格为平台无关数据核心。

建议：
- 去掉不必要的 RN 耦合
- transport 可注入
- 缓存/重试/限流一等化
- normalize 工具系统化
- provider 化导出，而不只是 API class

V3 卖点：
- 可单独当高德数据 SDK 用
- 可给 `core`/`navigation`/服务端共享

**建议的 breaking changes**
这些变化适合放到 `v3`，因为值得一次性打破兼容。

1. 初始化方式统一
旧：
`ExpoGaodeMapModule.initSDK()`

新目标：
内部统一 runtime，外部保留兼容壳，但文档主推 runtime/config 写法。

2. 搜索与算路能力 provider 化
旧：
直接导出很多函数

新：
推荐 `provider + client` 模式

3. 统一类型导出位置
旧：
各包有重复或近似类型

新：
领域类型尽量从稳定入口导出

4. 统一错误对象结构
旧：
不同包错误风格可能不同

新：
统一 `code/type/message/cause/retryable`

5. 导航状态显式化
旧：
更多依赖方法调用和回调

新：
引入明确 controller/state

**迁移策略**
不要一次全砍，建议三阶段。

迁移细节示例可参考 `docs/v3-migration.md`。

**阶段 1：内部重构，不破外部 API**
目标：
- 建 `domain/contracts/shared-react`
- 把重复类型搬过去
- 把 shared overlays/hooks 搬过去
- `core`/`navigation` 内部接入
- 对外 API 尽量不变

这是风险最低、收益最高的一步。

**阶段 2：新增 provider 和 runtime API**
目标：
- 新增 `createGaodeRuntime`
- 新增 `createNativeSearchProvider`
- 新增 `createWebRouteProvider`
- 文档开始主推新接口
- 老接口保留但标注 deprecated

这一阶段开始建立 `v3` 新心智。

**阶段 3：正式移除旧入口或降级为兼容层**
目标：
- 收敛重复导出
- 收敛重复类型
- 让旧 API 成为薄兼容层
- 真正完成架构切换

**建议的实施顺序**
1. 先做 `domain`
2. 再做 `contracts`
3. 再抽 `shared-react`
4. 先接 `core`
5. 再接 `navigation`
6. 再把 `web-api` provider 化
7. 最后把 `search` provider 化
8. 最后统一初始化和文档

这个顺序最稳，因为：
- `core` 测试最多，最容易兜住回归
- `navigation` 依赖现实约束更多，适合后接
- `search/web-api` provider 化更像接口层升级，放后面更安全

**V3 最终对外心智**
用户理解应该变成：

- 需要地图：装 `expo-gaode-map`
- 需要导航地图：装 `expo-gaode-map-navigation`
- 需要原生搜索：额外装 `expo-gaode-map-search`
- 需要 HTTP 数据能力：额外装 `expo-gaode-map-web-api`

但这四个包背后共享：
- 同一套类型
- 同一套错误模型
- 同一套 route/aoi/poi 数据模型
- 尽量一致的地图用户层 API

**我建议你把 V3 的一句话卖点写成**
“统一架构，不统一二进制；统一用户心智，不牺牲包体积和 SDK 约束。”
